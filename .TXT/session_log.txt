1. PRD — PRODUCT REQUIREMENTS DOCUMENT
   File: docs/PRD.md
   - Defined the core problem: engineering teams generate large volumes of
     operational data (PRs, Jira tickets, sprints, reviews) but lack timely
     analysis tooling
   - Identified 4 target personas: Engineering Manager, Tech Lead, Scrum Master,
     Developer
   - Defined 5 features: Daily Digest (F-01), Anomaly Detection (F-02),
     Conversational Q&A (F-03), Fivetran Sync Awareness (F-04), Trend Reports
     (F-05)
   - Set success metrics: <24h stale PR detection, >90% anomaly catch rate,
     -40% standup time, 80%+ conversational query coverage, >99% Fivetran sync
     reliability
   - Listed explicit non-goals: no Jira writes, no performance reviews, no
     individual developer scoring

2. TRD — TECHNICAL REQUIREMENTS DOCUMENT
   File: docs/TRD.md
   - Established 4-layer architecture: Fivetran → BigQuery → ADK+Gemini →
     CLI/Slack/GUI
   - Defined all 10 agent tools: 3 Fivetran MCP tools + 7 BigQuery SDK tools
   - Documented 5 hard system prompt rules (staleness check mandatory,
     aggregate-only, anomaly severity required, time-filtered SQL, staleness
     warning prepend)
   - Chose Google ADK as agent framework for first-class Gemini function calling
   - Chose Gemini 1.5 Pro/Flash as LLM (Pro for digest generation, Flash for
     conversational turns)
   - Specified in-memory 10-turn conversation history as sufficient for v1

3. DATA SCHEMA — BIGQUERY SCHEMA SPECIFICATION
   File: docs/DATA_SCHEMA.md
   - Defined 5 Fivetran-managed tables (read-only): github_pull_requests,
     github_commits, github_reviews, jira_issues, jira_sprints
   - Defined 3 agent-managed tables (read-write): agent_digests,
     agent_anomalies, agent_logs
   - Designed 5 pre-deployed BigQuery views: v_open_prs_aged,
     v_commit_velocity_daily, v_ticket_age_by_status, v_sprint_burndown,
     v_metric_baselines
   - Established naming conventions: github_* and jira_* never modified by
     agent; agent_* are agent-owned
   - Specified that GitHub usernames are hashed in agent_digests to avoid PII

4. APP FLOW — APPLICATION FLOW DOCUMENTATION
   File: docs/APP_FLOW.md
   - Documented primary flow: 11-step user query path from CLI/Slack/GUI input
     to agent response + audit log entry
   - Documented secondary flow: 9-step daily digest generation via Cloud Run
     job or local cron at 08:00
   - Documented tertiary flow: 6-step ad-hoc sync trigger when connector is
     stale and user confirms they want fresh data
   - Documented 5 error conditions and agent behaviour for each: MCP
     unreachable, BQ timeout, Gemini rate limit, no data found, sync failure

2. BIGQUERY SCHEMA (Phase 1.2 + 1.3 + 1.4)
   Files: schema/tables.sql, schema/views.sql, schema/deploy_schema.py
   - tables.sql: CREATE TABLE IF NOT EXISTS for agent_digests, agent_anomalies,
     agent_logs using {project}/{dataset} substitution placeholders
   - views.sql: CREATE OR REPLACE VIEW for all 5 views. v_open_prs_aged joins
     github_pull_requests + github_reviews with ROW_NUMBER() window to get latest
     review per PR. v_sprint_burndown uses SAFE_DIVIDE for null-safe pct_complete.
     v_metric_baselines UNIONs commit, PR age, and ticket age metrics for a
     single-table baseline lookup.
   - deploy_schema.py: reads both SQL files, strips SQL comments, splits on
     semicolons, substitutes placeholders, executes each statement via BQ client.
     Creates dataset with exists_ok=True. Prints per-statement [OK]/[ERR].

3. FIVETRAN TOOLS (Phase 2)
   File: tools/fivetran_tools.py
   - Uses Fivetran REST API v1 directly (vs Fivetran MCP Node.js server).
     Decision: avoids requiring Node.js running separately during hackathon.
     Functions are named identically to TRD spec; swappable to MCP later.
   - check_connector_status(): GET /connectors/{id}. Computes is_stale from
     last_successful_sync_at > 4h. Returns { status: "unreachable", is_stale: True }
     on any exception — agent continues with cached BQ data.
   - trigger_sync(): POST /connectors/{id}/force, polls every 30s, 300s timeout.
   - list_connectors(): GET /connectors, returns all with is_stale computed.
   - All functions check MOCK_MODE first and return from mock/fixtures.py.

4. BIGQUERY TOOL LAYER (Phase 3)
   Files: tools/bq_client.py, tools/bq_tools.py, mock/fixtures.py, mock/mock_client.py
   - bq_client.py: lazy singleton. Returns MockBigQueryClient if MOCK_MODE=true,
     real bigquery.Client otherwise. reset_client() for test isolation.
   - bq_tools.py: 7 ADK tool functions + write_agent_log (internal, not a tool).
     All MOCK_MODE short-circuits at function top — bq_client never called.
     All real-BQ paths use parameterised queries (@param syntax) for injection safety.
     query_sprint_health() computes expected_pct_complete and is_on_track in Python
     (not SQL) for portability.
     detect_anomalies() writes each anomaly to agent_anomalies via _write_anomaly().
     write_agent_log() is fire-and-forget: swallows exceptions so logging never
     breaks the query response path.
   - mock/fixtures.py: Sprint 22 (at-risk, 42.6% complete with 7 days left),
     4 open PRs (2 stale >3d), 14 days of commit history (2 repos), stale tickets
     per status, 2 active anomalies (medium PR age, high ticket_age_in_progress).
   - mock/mock_client.py: raises RuntimeError if called directly — makes bugs
     in the mock short-circuit logic immediately visible.

5. AGENT CORE (Phase 4)
   Files: agent/system_prompt.py, agent/agent.py, agent/main.py
   - system_prompt.py: SYSTEM_PROMPT encodes all 5 TRD rules with plain-English
     examples. Rule 3 (no individual metrics) includes an example of what the agent
     should say when asked about a specific developer.
   - agent.py: create_agent() registers all 10 tools. Model from GEMINI_MODEL env
     (default: gemini-1.5-flash). create_runner() wraps with InMemorySessionService
     and Runner. session_service is async (ADK 1.x).
   - main.py: MODE=api → uvicorn on API_HOST:API_PORT. MODE=cli → asyncio.run(run_cli).

6. DAILY DIGEST JOB (Phase 5)
   Files: jobs/daily_digest.py, jobs/slack_notifier.py
   - daily_digest.py: 9-step flow per APP_FLOW.md §3.2. Uses google.generativeai
     directly (not ADK) for synthesis — avoids initialising a full agent session
     just for batch generation. Falls back to _template_narrative() if Gemini fails.
   - slack_notifier.py: posts to SLACK_WEBHOOK_URL via httpx. Silent no-op if
     URL not set; logs warning on HTTP failure but does not raise (digest must
     complete even if Slack is unavailable).

7. INTERFACE LAYER (Phase 6)
   Files: interface/api.py, interface/cli.py
   - api.py: 8 FastAPI endpoints. Lifespan context manager initialises ADK agent

     in request body; InMemorySessionService keeps history per session_id.
     InMemorySessionService methods are async (get_session, create_session await-ed).
     CORS middleware open (allow_origins=["*"]) — tighten for production.
   - cli.py: async REPL. ANSI colour codes for DevPulse/You/Thinking labels.
     "Thinking…" text cleared on response. Same await pattern as api.py for session
     management.

--------------------------------------------------------------------------------
DESIGN DECISIONS / NOTES
--------------------------------------------------------------------------------

- Fivetran REST API used instead of Fivetran MCP Node.js server. Same data,
  simpler setup (no Node.js process). Tool function signatures unchanged from TRD.
  MCP can be substituted by replacing the httpx calls in fivetran_tools.py with
  ADK MCPToolset calls.

- MOCK_MODE implemented at tool function level (not client level). Each function
  has an early return from fixtures.py. The ADK agent and Gemini still run
  normally in mock mode — only the data layer is mocked. This lets the full
  agent reasoning pipeline be tested without any cloud credentials.

- InMemorySessionService.get_session() and create_session() are async in ADK 1.x.
  Both interface/api.py and interface/cli.py await these calls.

- Session history is per session_id passed in the request body. No server-side
  session lookup by user. Frontend is responsible for persisting session_id.

- write_agent_log() is not registered as an ADK tool. It is called directly by
  the interface layer after each turn. This keeps the tool registry clean and
  avoids Gemini ever trying to call it.

- Daily digest uses google.generativeai directly (not ADK). This avoids the
  overhead of ADK session management for a batch job with no conversation history.
