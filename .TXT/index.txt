  5. DATABASE SCHEMA
────────────────────────────────────────────────────────────────────────────────

  Dataset: devpulse (Google BigQuery)
  Tables prefixed github_ / jira_ are Fivetran-managed (read-only).
  Tables prefixed agent_ are agent-managed (read-write).

  ── Fivetran-Managed Tables (read-only) ──────────────────────────────────────

  TABLE: github_pull_requests
    Stores all PR metadata synced from GitHub. Fivetran writes; agent reads.
    RLS: No row-level security in BigQuery; access controlled via service acct.
    Index: id (primary key), state, created_at (used in all time-filtered queries)

    id                  STRING      GitHub PR ID (primary key)
    repo_name           STRING      Repository full name (org/repo)
    number              INTEGER     PR number within the repo
    title               STRING      PR title
    state               STRING      open | closed | merged
    author_login        STRING      GitHub username of the PR author
    created_at          TIMESTAMP   When the PR was opened
    updated_at          TIMESTAMP   Last activity on the PR
    merged_at           TIMESTAMP   When merged (NULL if not merged)
    closed_at           TIMESTAMP   When closed (NULL if open)
    review_requested_at TIMESTAMP   When first review was requested
    _fivetran_synced    TIMESTAMP   Last time Fivetran synced this row

  TABLE: github_commits
    Stores commit history synced from GitHub. Agent reads for velocity metrics.
    RLS: None. Index: sha (primary key), author_login, committed_at.

    sha                 STRING      Commit SHA (primary key)
    repo_name           STRING      Repository full name
    author_login        STRING      GitHub username of committer
    committed_at        TIMESTAMP   Commit timestamp
    additions           INTEGER     Lines added
    deletions           INTEGER     Lines deleted
    message_summary     STRING      First line of commit message
    _fivetran_synced    TIMESTAMP   Last Fivetran sync timestamp

  TABLE: github_reviews
    Stores PR review events. FK to github_pull_requests.id.
    RLS: None. Index: id (primary key), pull_request_id, submitted_at.

    id                  STRING      Review ID (primary key)
    pull_request_id     STRING      FK → github_pull_requests.id
    reviewer_login      STRING      GitHub username of reviewer
    state               STRING      APPROVED | CHANGES_REQUESTED | COMMENTED
    submitted_at        TIMESTAMP   When the review was submitted
    _fivetran_synced    TIMESTAMP   Last Fivetran sync timestamp

  TABLE: jira_issues
    Stores Jira ticket data. FK sprint_id → jira_sprints.id.
    RLS: None. Index: id (primary key), status, sprint_id, updated_at.

    id                  STRING      Jira issue ID (primary key)
    key                 STRING      Issue key (e.g. ENG-123)
    project_key         STRING      Jira project identifier
    summary             STRING      Issue title / summary
    issue_type          STRING      Story | Bug | Task | Epic | Sub-task
    status              STRING      To Do | In Progress | In Review | Done | Blocked
    priority            STRING      Highest | High | Medium | Low | Lowest
    assignee_id         STRING      Jira account ID of assignee
    reporter_id         STRING      Jira account ID of reporter
    created_at          TIMESTAMP   Issue creation time
    updated_at          TIMESTAMP   Last status change or update
    resolved_at         TIMESTAMP   Resolution time (NULL if unresolved)
    sprint_id           STRING      FK → jira_sprints.id (NULL if backlog)
    story_points        FLOAT       Estimate in story points
    _fivetran_synced    TIMESTAMP   Last Fivetran sync timestamp

  TABLE: jira_sprints
    Stores sprint metadata. Referenced by jira_issues.sprint_id.
    RLS: None. Index: id (primary key), state, start_date, end_date.

    id                  STRING      Sprint ID (primary key)
    name                STRING      Sprint name (e.g. 'Sprint 42')
    board_id            STRING      FK to the Jira board
    state               STRING      active | closed | future
    start_date          DATE        Sprint start date
    end_date            DATE        Sprint planned end date
    completed_date      DATE        Actual completion date (NULL if ongoing)
    _fivetran_synced    TIMESTAMP   Last Fivetran sync timestamp

  ── Agent-Managed Tables (read-write) ────────────────────────────────────────

  TABLE: agent_digests
    Stores daily engineering health summaries written by the agent.
    One record per business day. narrative field contains Gemini output.
    Index: digest_id (primary key), digest_date.

    digest_id           STRING      UUID primary key
    generated_at        TIMESTAMP   When this digest was produced
    digest_date         DATE        The business day this digest covers
    open_pr_count       INTEGER     Total open PRs at digest time
    stale_pr_count      INTEGER     PRs open >3 days without activity
    commit_count_7d     INTEGER     Commits in the past 7 days
    commit_count_prev_7d INTEGER    Commits in the 7 days before that
    anomaly_count       INTEGER     Number of anomalies detected
    anomaly_summary     STRING      JSON array of top anomalies
    sprint_on_track     BOOLEAN     Whether the active sprint is on track
    narrative           STRING      Agent-generated natural-language summary
    model_used          STRING      Gemini model version used

  TABLE: agent_anomalies
    Stores every detected metric anomaly. Severity: low/medium/high based on
    z-score magnitude. resolved_at populated when metric returns to baseline.
    Index: anomaly_id (primary key), metric_name, detected_at, severity.

    anomaly_id          STRING      UUID primary key
    detected_at         TIMESTAMP   When the anomaly was flagged
    metric_name         STRING      e.g. pr_age_days, commit_count
    metric_value        FLOAT       Observed value at detection time
    baseline_mean       FLOAT       Rolling 30-day mean for this metric
    baseline_stddev     FLOAT       Rolling 30-day standard deviation
    z_score             FLOAT       Deviation magnitude (|z| > 2 = anomaly)
    severity            STRING      low | medium | high
    entity_type         STRING      repo | ticket | sprint
    entity_id           STRING      Specific repo/ticket/sprint affected
    resolved_at         TIMESTAMP   When metric returned to normal (NULL if ongoing)

  TABLE: agent_logs
    Audit trail of every agent interaction. One record per conversation turn.
    Used for cost tracking (bq_bytes_processed) and debugging.
    Index: log_id (primary key), session_id, logged_at.

    log_id              STRING      UUID primary key
    session_id          STRING      Groups all turns in one conversation
    logged_at           TIMESTAMP   Timestamp of the interaction
    user_query          STRING      Raw user input
    tools_called        STRING      JSON array of tool names invoked
    bq_bytes_processed  INTEGER     Total BigQuery bytes billed for this turn
    latency_ms          INTEGER     Total wall-clock time in milliseconds
    model_used          STRING      Gemini model version
    response_preview    STRING      First 500 chars of agent response

  ── BigQuery Views ────────────────────────────────────────────────────────────

  VIEW: v_open_prs_aged
    Open PRs with age in days, bucketed. Joins github_pull_requests +
    github_reviews. Used by query_open_prs tool.

  VIEW: v_commit_velocity_daily
    Daily commit counts per repo. Derived from github_commits.
    Used by query_commit_velocity tool.

  VIEW: v_ticket_age_by_status
    Ticket age grouped by current status. Derived from jira_issues.
    Used by query_stale_tickets tool.

  VIEW: v_sprint_burndown
    Story points completed vs remaining by day. Joins jira_issues +
    jira_sprints. Used by query_sprint_health tool.

  VIEW: v_metric_baselines
    Rolling 30-day mean + stddev per metric. Joins agent_anomalies,
    github_commits, jira_issues. Used by detect_anomalies tool.

  6. API ROUTES
────────────────────────────────────────────────────────────────────────────────

  No HTTP API routes exist yet. The planned interface is:

  POST  /query          (Slack webhook / GUI mode)
    Accepts a JSON body { "text": "<user query>", "session_id": "<id>" }.
    Runs a full agent turn: check_connector_status → route to BQ tools →
    synthesise → write_digest if applicable → return response.
    Auth: Slack signing secret validation (HMAC-SHA256 header check).
    Non-obvious: session_id groups turns for in-memory history lookup.

  POST  /digest/trigger  (Cloud Run job or admin call)
    Triggers an immediate daily digest run outside the 08:00 schedule.
    No request body required. Returns the generated digest_id on success.
    Auth: Requires a service account token or a shared secret header.

  GET   /health          (Cloud Run liveness probe)
    Returns 200 OK with { "status": "ok", "timestamp": "<ISO8601>" }.
    No auth required. Used by Cloud Run health checks.

  None of these routes are implemented yet.

  9. ARCHITECTURE NOTES
────────────────────────────────────────────────────────────────────────────────

  ── AI PIPELINE ──────────────────────────────────────────────────────────────

  User query (CLI/Slack/GUI)
    → ADK Agent receives query, initialises turn
    → Gemini evaluates query against system prompt rules
    → check_connector_status (GitHub) via Fivetran MCP
    → check_connector_status (Jira) via Fivetran MCP
    → Staleness check: if >4h old, prepend warning to response
    → Route to 1..N BigQuery tools (parameterised SQL)
    → BigQuery executes queries, returns structured rows
    → Gemini synthesises rows into natural-language answer
    → If new insight: write_digest → agent_digests table
    → Return answer to user
    → Logger writes full interaction → agent_logs table

  ── DAILY DIGEST FLOW ────────────────────────────────────────────────────────

  Cloud Run / cron triggers at 08:00
    → list_connectors (confirm all sources active)
    → detect_anomalies (all metric categories) → agent_anomalies table
    → query_open_prs (3-day threshold)
    → query_commit_velocity (7-day window)
    → query_sprint_health (active sprint)
    → Gemini synthesises all results into structured digest
    → write_digest → agent_digests table
    → Optional: POST digest to Slack webhook

  ── DATA FRESHNESS & STALENESS ───────────────────────────────────────────────

  Fivetran syncs GitHub + Jira to BigQuery on a 1-hour cadence.
  Every agent turn begins with a staleness check via the MCP server.
  If connector last_sync > 4 hours ago:
    Agent prepends: "Warning: [source] data is N hours old."
  If user requests a fresh sync:
    trigger_sync → poll check_connector_status every 30s → timeout at 5 min
    → proceed with original query once sync confirmed complete.
  Fallback if MCP unreachable:
    Answer from cached BQ data; warn user sync status is unavailable.

  ── SECURITY MODEL ───────────────────────────────────────────────────────────

  All secrets in environment variables; never in source code or version control.
  FIVETRAN_API_KEY scoped to read + trigger-sync only (not admin).
  BigQuery service account scoped to devpulse dataset only (no project-wide).
  GitHub usernames hashed in agent_digests to avoid storing PII.
  Slack webhook URL stored in GCP Secret Manager when deployed to Cloud Run.
  Individual developer performance metrics never exposed — aggregate only.
  All BigQuery queries parameterised to prevent injection via user input.
